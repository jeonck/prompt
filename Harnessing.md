솔직히 저는 지난주까지 하네스 불신론자였습니다. 이 이야기를 몇몇 관계자 분들 앞에서도 이야기했습니다. 사실 팀에서 일할 땐 필요를 못 느꼈거든요. 동료가 실수를 잡아주니까. 

혼자 움직이기 시작하니까 상황이 달라졌습니다. 에이전트가 틀려도 잡아줄 시스템이 없으면 1인이 뭔가를 제대로 만드는 게 불가능합니다.

그래서 읽은 하네스 엔지니어링 관련 좋은 글 5개를 추려 안내드립니다.
어제 글 하나가 추가적으로 게시되어서 5개의 아티클이 되었습니다.

1️⃣ The Anatomy of an Agent Harness. 하네스가 뭔지 이 글 하나로 끝납니다

blog: https://lnkd.in/g225NR5N

Agent = Model + Harness. 모델이 아닌 모든 코드와 설정과 실행 로직이 하네스입니다. 같은 Opus 4.6 모델인데 어떤 하네스에 넣느냐에 따라 벤치마크 점수가 크게 달라져요. 읽고 나서야 프롬프트 엔지니어링과 하네스 엔지니어링의 경계가 잡혔습니다.

• 모델은 텍스트를 입출력할 뿐이고 상태 관리와 도구 실행은 전부 하네스 영역 
• LLM 을 반복 루프에 넣는 것 자체가 이미 하네스

2️⃣ Improving Deep Agents. 모델 안 바꾸고 13.7점 올린 방법

blog: https://lnkd.in/gDDvATPJ

하네스 회의론을 깨는 결정타였습니다. LangChain 팀이 모델을 고정한 채 하네스만 바꿔서 Terminal Bench 2.0 점수를 52.8%에서 66.5%로 끌어올렸어요. Top 30 밖이던 에이전트가 Top 5 에 진입한 겁니다.

• 같은 파일을 10번 넘게 고치는 무한 루프를 자동 탐지 
• 에이전트 종료 전에 검증 단계를 강제로 끼워 넣음

3️⃣ How Middleware Lets You Customize Your Agent Harness. 커스터마이징 진입장벽은 훅 3개입니다

blog: https://lnkd.in/gkyJpKWs

통념을 하나 뒤집겠습니다. 하네스를 잘 쓰려면 프레임워크를 깊이 알아야 한다고 생각하기 쉬운데 실제로는 시작 전 준비와 모델 호출 전후 처리와 호출 감싸기 세 가지 훅이면 대부분 커버됩니다.

• 파일 시스템으로 맥락을 저장하고 불러오기 
• 서브에이전트를 격리해서 실행하기 
• 긴 작업에서 맥락이 넘칠 때 자동 축약하기

4️⃣ How we build evals for Deep Agents. 평가 항목은 많이 만드는 게 아니라 잘 고르는 겁니다

blog: https://lnkd.in/guExjNni

가장 인상적이었던 문장이 있습니다. 평가 항목을 수백 개 쌓는다고 에이전트가 좋아지지 않는다는 거예요. 

항목 하나하나가 시스템 행동을 바꾸는 방향타이기 때문에 무작정 늘리면 프로덕션과 동떨어진 곳으로 최적화됩니다.

• 정답률과 경로 효율을 분리해서 따로 측정 
• 더 이상 신호를 주지 않는 항목은 정기적으로 정리

5️⃣ Better Harness. 평가 결과로 하네스를 자동 개선하는 루프

blog: https://lnkd.in/g5UBPUKE

4번이 평가를 어떻게 만드느냐였다면 이 글은 그 평가를 신호 삼아 하네스를 자동으로 개선하는 방법입니다. 

과적합 방지를 위해 학습용 셋과 검증용 셋을 나누는 구조가 머신러닝 훈련 파이프라인과 똑같아요.

• 실패 기록을 분석해서 하네스 변경 방향을 자동 제안 
• 통과한 항목을 회귀 테스트로 전환해서 퇴보 방지
